## 1. 목적과 배경

OPOD의 AI 액션(채팅 답장, 향후 이미지 생성·통화 등)은 호출당 외부 API 원가가
발생한다. 이 문서는 그 원가를 회수하는 종량제 크레딧 정책을 정의한다.

핵심 설계 질문은 다음이었다:

> 대화가 길어질수록 매 턴 전체 히스토리가 입력 토큰으로 재전송되어 원가가
> 증가한다. 이 원가 증가를 "토큰 사용량 기반 가변 과금"으로 유저에게
> 전가할 것인가?

§2에서 시장 조사를 근거로 이 질문에 답한다.

## 2. 과금 모델 결정: 왜 토큰 종량제가 아닌 정액 종량제인가

### 2.1 두 가지 선택지

| | (A) 토큰 가변 과금 | (B) 정액 + 원가 엔지니어링 |
|---|---|---|
| 원리 | 실제 토큰 사용량만큼 차감 | 턴당 고정가, 컨텍스트를 서버가 상한 |
| 원가 정합성 | 완벽 | 상한 설계로 보장 |
| 유저 예측 가능성 | **없음** — 같은 행동의 가격이 대화 길이에 따라 변함 | 완벽 |
| 제품 정합성 | 긴 대화(=핵심 리텐션 루프)를 벌점화 | 긴 대화를 장려 |

### 2.2 시장 조사 결과 (2026-07, 상세·출처는 부록 §10)

- **토큰/컨텍스트 연동 가변 과금을 쓰는 소비자 챗 제품은 사실상 Poe 하나**이고,
  Poe는 카테고리에서 가장 과금 불만이 많은 제품이다 ("메시지 하나에 포인트가
  얼마 나갈지 모른다"). Poe조차 컨텍스트 자동 절삭(auto-manage context)을
  기본값으로 켜서 비용을 묶는다.
- **국내 유일 대규모 하이브리드는 크랙(뤼튼)**: 턴당 모드별 정액 + 출력
  800토큰 초과분을 100토큰 단위로 추가 과금. 불투명한 가격/출력량 변경으로
  2025년 유저 온라인 시위가 2회 발생했고, 커뮤니티에 "원/턴" 가격 감시
  문화가 생겼다.
- **나머지 전원(Character.AI, 제타, 러비더비, NovelAI, AI Dungeon,
  SpicyChat, Replika...)은 정액 + 원가 엔지니어링**: 컨텍스트 윈도우 캡,
  요약/메모리 시스템, 저가 모델 라우팅으로 긴 대화의 원가를 묶고 유저에게는
  고정 단가만 노출한다. 컨텍스트 자체는 상위 상품으로 판다(SpicyChat
  4K/8K/16K 티어, AI Dungeon "추가 컨텍스트 1크레딧/액션").
- 업계 공통 규칙: **전송 전에 다음 메시지의 가격을 알 수 있어야 한다.**
  이를 어긴 사례(Poe 가변 포인트, 크랙 잠수함 패치, Character.AI Charms)가
  전부 카테고리 최대 반발 사례다.

### 2.3 결정

**(B) 정액 종량제를 채택한다.**

1. 모든 과금 액션은 **실행 전 고지된 고정 크레딧**을 차감한다.
2. 대화가 길어질 때의 원가 증가는 과금이 아니라 **서빙 정책(§2.4)** 으로 막는다.
3. 향후 사용량을 가격에 반영해야 하면 **유저가 선택하는 모드별 정액**
   (예: 기본 2 / 장문 5 / 프리미엄 10크레딧 — 러비더비 3/5/10잼 패턴)으로
   확장한다. 가변이되, 전송 전에 가격이 보이는 구조만 허용한다.
4. **금지 사항**: 숨은 토큰 과금, 대화 길이에 따라 몰래 변하는 단가,
   고지 없는 출력량·단가 변경 (크랙 반발 사례의 교훈).

### 2.4 컨텍스트 상한 정책 (원가 통제 장치)

정액제가 성립하려면 턴당 원가에 상한이 있어야 한다.

- 기본 채팅의 턴당 입력 토큰 상한: **~4K 토큰** (출력 ~300 토큰).
- 컨텍스트 구성(히스토리 도입 시): 시스템 프롬프트 + 캐릭터 프로필 +
  대화 요약 메모리 + 최근 N턴 롤링 윈도우. 상한 초과분은 요약으로 압축.
- 현행 구현은 히스토리 없이 단일 메시지만 전송하므로
  (`src/domain/messages/message-reply.provider.ts`) 상한을 자연 충족한다.
  **대화 히스토리/페르소나를 컨텍스트에 추가하는 순간부터 이 상한이
  구속력을 가진다.**
- 상한을 넘는 고급 경험(더 긴 기억, 고급 모델)은 단가를 올리는 게 아니라
  **별도 프리미엄 모드 상품**으로 분리한다.

## 3. 재화 정의

- 명칭: **크레딧** (credit). 정수 단위, 소수점 없음 (현행 원장 규칙과 동일).
- 기준가: **1크레딧 = ₩10** (웹 결제 기준). 제타 피스(10원)와 동일 앵커로,
  턴당 1~50크레딧 범위의 액션 가격을 표현하기 적절한 입도다.
- **재앵커 주의**: 현행 코드의 `credits_100`(100크레딧 = ₩9,900, 1크레딧 ≈
  ₩99)은 placeholder이며 실유저·실결제가 없으므로 이 정책으로 폐기한다.
  ₩99/크레딧은 시장 챗 턴당 실효가(2~85원) 대비 과가격이다.
- 크레딧 구분:
  - **유상 크레딧**: 결제로 취득. 유효기간 없음.
  - **무상 크레딧**: 가입 보너스·출석·이벤트·CS 보상. **지급일로부터 30일 유효**
    (국내 관행: 크랙 30일).

## 4. 액션 단가표

원가 추정 전제: 환율 1$ ≈ ₩1,400. 기본 채팅은 §2.4 컨텍스트 캡(입력 ~4K /
출력 ~300 토큰) 적용 기준.

| 액션 | 크레딧 | 판매가 | 추정 원가 | 원가 근거 | 배수 |
|---|---:|---:|---|---|---|
| `chat_reply` (기본) | 2 | ₩20 | ₩1–8 | 저가 모델(Flash급 $0.1/$0.4 per MTok) ≈ ₩0.7, 중급(Haiku급 $1/$5) ≈ ₩8 | 2.5–20× |
| `chat_reply` (프리미엄 모드, 향후) | 5–10 | ₩50–100 | ₩25–45 | 상위 모델(Sonnet급 $3/$15), 컨텍스트 8K + 장문 출력 600토큰 ≈ $0.033 | ~2–3× |
| `image_generate` (향후) | 50 | ₩500 | ₩30–110 | 이미지 API(Flux/DALL-E급 $0.02–0.08/장); 자체 GPU(ComfyUI) 시 ₩10–30 | 5–15× |
| `story_generate` (향후) | 30 | ₩300 | ₩20–60 | LLM 장문 생성(출력 ~2.5K 토큰, Sonnet급 ≈ $0.044; 저가 모델 ₩10–20) | 5–15× |
| `call_turn` (향후) | 15 | ₩150 | ₩20–80 | STT(Whisper $0.006/분, 30초 ≈ ₩4) + LLM ₩5–10 + TTS(OpenAI급 ₩6 ~ ElevenLabs급 ₩40+/300자) | 2–7× |

가격 원칙:

1. 액션 단가는 **추정 원가의 최소 2.5–4배**를 유지한다 (마진 + 무료 지급 재원).
2. 원가가 이 배수를 깨는 모델 조합(예: 기본 채팅에 Sonnet급)은 기본 모드에
   쓰지 않고 프리미엄 모드로 분리한다. **단가표가 모델 선택을 제약한다.**
3. 단가는 전부 서버 설정값으로 관리하며, 변경 시 §8.4 고지 절차를 따른다.
4. 시장 위치: 기본 채팅 ₩20/턴은 국내 실효가 범위(러비더비 찰떡 ~₩20–30,
   크랙 파워챗 ₩20, Opus급 ₩85)의 하단이다.

  웹 충전으로 유도한다 (제타 +40%, 네이버웹툰 +20% 등 국내 표준 관행).
- 참고: 구글의 결제 정책 개편(수수료 인하 + 아웃링크 결제 허용)이 한국에는
  2026-12 적용 예정 — 시행 후 앱 내 할증 폭 재검토.
- 유상 크레딧은 유효기간 없음. 환불은 §8.1.

## 6. 무료 크레딧: 가입 보너스 + 출석체크

종량제에서 신규 유저가 결제 전에 제품 가치를 경험할 수 있는 유일한 통로다.

### 6.1 가입 보너스

- 가입 완료 시 **100크레딧** 1회 지급 (₩1,000 상당 = 기본 채팅 50턴).
- 유효기간 30일.

### 6.2 출석체크

- 하루 1회 **10크레딧** (기본 채팅 5턴). 날짜 경계는 **KST 00:00**.
- 월 단위 출석판(매월 1일 리셋), 누적 출석 보너스:
  - 월 7일차 +20 / 14일차 +30 / 30일차(개근) +50크레딧
- 월 최대 무상 지급: 300(일일) + 100(보너스) = **400크레딧
  (₩4,000 상당, 원가 환산 ₩200–1,600/월/유저)** — 지속 가능한 수준.

### 6.3 공통 규칙

- 무상 크레딧 유효기간: **지급일 + 30일** (축적 방지).
- 차감 순서: **소멸 임박 무상 → 무상 → 유상** 순으로 소진.
- 어뷰징 방지: 계정당 1일 1회(서버 기준), 향후 필요 시 디바이스 지문·본인
  인증 조건 추가. 다계정 대량 가입 보너스 어뷰징은 가입 시점 제한
  (이메일 인증 등)으로 대응.

## 7. 차감 흐름: reserve → capture / release

아키텍처 문서의 원칙("The ledger records final grants and debits")을 따른다.
예약은 원장 밖의 별도 상태로 관리하고, 원장에는 확정분만 남긴다.

```text
액션 요청
  -> reserve   : 가용 잔액 확인 후 정가만큼 예약 (부족 시 즉시 거절)
  -> 액션 실행  : LLM 호출 / 생성 잡 등
  -> capture   : 성공 -> 예약 확정, 원장에 debit 기록
  -> release   : 실패·타임아웃 -> 예약 해제, 원장 기록 없음 (과금 안 됨)

상태: reserved -> captured | released   (종결 상태에서 재전이 불가)
```

규칙:

1. **가용 잔액** = 원장 잔액 − 활성(reserved) 예약 합계. reserve는 가용
   잔액 기준으로 판정한다.
2. **원자성**: 잔액 확인과 예약 생성은 단일 트랜잭션. 동시 요청이 잔액을
   초과 예약할 수 없어야 한다.
3. **멱등성**: 예약은 액션 참조(예: `chat_reply:<message_id>`)와 결부하고
   유니크 제약으로 중복 예약·중복 capture를 차단한다.
4. **타임아웃**: 예약은 액션별 TTL(기본 채팅 5분) 경과 시 자동 release.
   유저 크레딧이 실패한 요청에 잠기지 않아야 한다.
5. **정액제이므로 예약액 = 확정액.** 향후 가변 요소(통화 분당 과금 등)가
   생기면 최대치 예약 → 실사용 capture 패턴을 사용한다.
6. **잔액 부족**: 액션 시작 전 차단하고 `402` + 에러 코드
   `INSUFFICIENT_CREDITS`로 응답, 클라이언트는 충전 화면으로 유도한다.
   유저 메시지 저장 등 어떤 부수효과도 남기지 않는다.

## 8. 운영 정책

### 8.1 환불·회수

- 유료 구매 환불(결제 취소) 시 해당 구매로 지급된 크레딧을 회수(debit)한다.
  잔액이 회수량보다 적으면 잔액 전액 회수 후 차액은 환불액에서 공제한다.
- 서비스 장애로 액션이 실패했는데 capture된 경우: 확인 즉시 보상 grant.
  (release 누락은 버그로 취급, 정산 리포트로 탐지 — opod-admin reconciliation.)

### 8.2 관리자 수동 지급

- 기존 opod-admin `POST /credits/grants`의 용도: CS 보상, 이벤트 지급.
- 수동 지급도 무상 크레딧으로 분류(30일 유효)하되, 사유(reason) 기록 필수.

### 8.3 향후 후보 (이번 정책 범위 밖, 기록만)

- **구독 패스**: 광고 제거 + 월 크레딧 번들 (제타패스: 웹 ₩10,900/월 참고).
- **광고 리워드**: 광고 시청 → 소량 무상 크레딧 (제타 매출의 ~60%가 광고).
- **크리에이터 정산**: 유저 제작 캐릭터에서 소모된 유상 크레딧의 일정률
  정산 (크랙 1.5% 사례).

### 8.4 가격 변경 절차

- 단가·패키지·무료 지급량 변경은 **최소 7일 전 공지**하고 변경 이력을
  이 문서에 남긴다. 무고지 변경은 금지 (크랙 잠수함 패치 → 유저 시위 사례).

### 10.3 사례 분석: 토큰 가변 과금의 실패 패턴

- **Poe**: 같은 질문도 대화가 길수록, 모델이 좋을수록 포인트가 더 나간다.
  유저 불만의 핵심은 금액이 아니라 **예측 불가능성**. 무료 포인트 90% 삭감
  (3,000→300/일)과 겹쳐 앱스토어 평점 급락.
- **크랙**: "기본요금 + 초과 종량"은 이론상 합리적이나, 출력량을 몰래 줄이는
  잠수함 패치가 결합되자 "과금을 숨긴다"는 불신으로 폭발 (2025-02, 03
  온라인 시위). 교훈: 가변 요소가 있는 순간 유저는 가격 감시자가 되고,
  투명성 비용이 급증한다.
- **일반화**: 소비자 제품에서 토큰은 유저의 멘탈 모델에 없는 단위다.
  "메시지 1개 = N크레딧"은 이해되지만 "토큰 1,000개당 N원"은 불신을 만든다.
